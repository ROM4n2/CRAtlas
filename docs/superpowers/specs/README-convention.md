# README 撰写规范（CRAtlas 项目）

## 目的

每个源代码目录应有 `README.md`，降低新成员上手成本，便于 code review 和后期维护。

## 必须包含的章节

| 章节 | 说明 |
|------|------|
| `# 目录名/` | 顶部一级标题，以 `/` 结尾，如 `# components/map/` |
| `## 职责` | 一句话概括本目录的功能定位 |
| `## 文件清单` | 表格形式列出每个文件/子目录及其职责 |
| `## 数据流 / 依赖关系` | 说明本目录内组件的输入输出、依赖方向 |
| `## 使用方式` | 代码示例（如果适用）|
| `## 约定` | 命名、注释、性能、架构规则等注意事项 |

## 写作原则

1. **语言**：整个项目统一使用 **中文**（与 CLAUDE.md 项目指令一致）。根目录 `README.md` 可保留英文。
2. **表格优先**：文件清单用 Markdown 表格，表头为 `文件` | `职责`。
3. **代码示例**：使用 TypeScript/TSX，带语言标记的 fenced code block。
4. **数据流图**：使用 ```` ``` ``` ` 纯文本图或 ```````mermaid`（如果复杂）。
5. **跨引用**：引用 `lib/`、`data/` 等其他目录的 README 用相对路径。
6. **TODO 标记**：尚未实现但在计划中的功能用 `<!-- TODO: ... -->` 标注，不写入可见正文。
7. **不重复**：不复制 `CLAUDE.md` 中已有的全局规范。只写本目录特有的内容。

## 示例头部

```markdown
# components/map/

## 职责

地图可视化组件——ECharts 中国地图渲染、图例、事件标注。
```

## 审核标准

- ✅ 每个源文件在文件清单中有对应行
- ✅ 职责清晰、边界明确
- ✅ 数据流方向正确（符合四层架构）
- ❌ 不包含全局规范（已在 CLAUDE.md 中）
- ❌ 不复制代码注释（代码本身应自文档化）
