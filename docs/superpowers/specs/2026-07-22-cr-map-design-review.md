## 审查结论（摘要）

文档整体结构清晰、技术选型合理、分阶段路线图可行。但存在 **3 个阻塞性问题**（核心可视化机制定义缺失、核心数据无采集路径、核心机制与意识形态目标冲突）和若干一致性问题。**在动笔写任何一行组件代码之前**，必须先解决 P0 问题，否则实现阶段会返工。

## P0 — 阻塞实现（不解决则 Phase 1 无法真正启动）

### Prompt 0.1 【核心可视化机制定义缺失】地图"省份着色"的数据基础未定义

- **原文位置**：§4.1 地图组件、§3.1 `RegionControl` / `RegionControlMap`、§6 `getRegionControl(date)`
- **问题**：文档定义了"某时间点某省被哪个派系控制"的数据结构 `RegionControl { factionId, strength }` 和查询函数 `getRegionControl(date)`，但**从未定义这个数据从哪来、怎么算、谁录入**。这是整个地图可视化的核心输入——没有它，地图着色就是空函数。当前数据层 `data/*.ts` 里没有 `regionControls.ts`，也没有任何关于"控制强度 0-1"如何量化的说明。"控制"本身是争议性判断（是派系掌权？有大规模组织？有武装？），不是客观事实直接可得。
- **修改要求**：
  1. 在数据层新增 `data/regionControls.ts`，定义按时间段的地区控制记录（需含 `startDate`/`endDate`，因为控制权会随时间变化）。
  2. 明确定义"控制"的操作性定义（operational definition）：什么算"某派系控制某省"？给出可判据（如：省级革委会/党委被该派系主导、该派系有大规模群众组织等）。
  3. 定义 `strength` 的量化规则（0-1 代表什么？是控制稳固度？还是势力占比？如何从史料推断？）。
  4. 承认数据稀疏性：不是每个省每个月都有明确控制方，需定义"无数据/争夺中/多方割据"等状态的表示方式。当前颜色方案里有"争夺中"和"无数据"，但数据结构 `RegionControl` 只支持"一个 factionId + 一个 strength"，**无法表达"多方割据"**——这是数据结构缺陷，一并修复。
  5. 在 §4.1 渲染逻辑里补充：当某地区处于"争夺中/多方割据"时，地图视觉如何表达（当前只写了按 factionId 单色着色，无法表现争夺）。

### Prompt 0.2 【核心数据无采集路径】"省份级派系控制"史料的可获得性未验证

- **原文位置**：§3.4、§7 Phase 3 "内容扩充"、§10 风险表
- **问题**：Prompt 0.1 的数据结构问题之上，还有一个更根本的**可获得性问题**：文革时期省级权力交替的派系归属，在公开史料中**既稀缺又高度争议**。文档把"史料来源争议"列为风险，但缓解措施只有"标注来源"——这不够。如果核心数据（省级控制时间线）根本找不到足够可靠的来源，地图可视化的前提就不成立。这不是技术问题是产品前提问题。
- **修改要求**：
  1. 在动笔 Phase 1 之前，先做一个**可行性预研（feasibility spike）**：选 3-5 个省份（如北京、上海、辽宁、广东、四川），实际去史料里查能否构建 1966-1976 逐季/逐年的控制时间线。把预研结果写进文档。
  2. 如果不可行，准备**降级方案**并在文档中明确：例如（a）只标注"事件发生地"而不着色全省，用事件散点代替省级着色；（b）只在有明确数据的省份着色，其余标"无数据"；（c）把粒度从"省"放宽到"主要城市/大区"。
  3. 在 §10 风险表里，把这条风险升级为最高优先级，并附上降级方案的触发条件（"如果 N 周内无法为至少 X 个省构建时间线，则启用降级方案 Y"）。
  4. 这是整个项目最大的产品风险，文档目前轻描淡写了。

### Prompt 0.3 【核心机制与意识形态目标冲突】"不声称价值中立"与"派系分类"的张力未解决

- **原文位置**：§0.2 意识形态立场、§5.4 关于页、§10 风险表
- **问题**：文档明确说"不声称价值中立"、"以马列毛主义为编纂立场"，但同时又要求"每条信息标注史料出处让用户自行判断"。这里有一个产品层面的**真实张力**：派系分类（rebel/conservative/military/other）本身就是价值判断——"造反派/保皇派"的标签本身就是立场。文档既要用这套分类作为核心组织框架，又要声明"区分立场与史料"，但没有说明**当史料本身冲突时怎么办**（比如同一事件，文革后官方档案 vs 学术史料 vs 当事人回忆录的派系定性可能完全不同）。
- **修改要求**：
  1. 在 §0.2 或新增 §0.6 中，明确"史料冲突"的处理策略：是取主流学术共识？是多说并存？还是按编纂立场取舍？给出可操作的规则。
  2. 定义"编纂立场介入的边界"：哪些环节立场优先（如分类框架、事件选择），哪些环节史料优先（如日期、地点、人名）。让读者一眼看出哪里是立场、哪里是史料。
  3. 在详情页（§5.4、§3.1）增加"争议标注"机制：当某条信息存在史料冲突时，在页面上显式标注"关于此事存在多种说法"并并列呈现，而不是只呈现一种。
  4. 这不是要你放弃立场，而是让立场**透明且一致**——否则"区分立场与史料"就是空话，专业研究者（次要受众）不会信任这个平台。

## P1 — 影响一致性与可执行性（不解决则开发中会反复卡住）

### Prompt 1.1 【数据结构重复 + 无约束】`Person.factionIds` 与 `Person.affiliations` 的同步无机制保障

- **原文位置**：§3.1 `Person` 接口、§3.2 字段使用边界
- **问题**：文档用 §3.2 定义了 `factionIds` 是冗余字段、修改 `affiliations` 时要同步更新 `factionIds`。但这只是文字约定，没有任何机制保障。TypeScript 编译期检查不出"这两个字段不一致"的 bug，运行时也不会报错。随着数据量增长（Phase 3+），几乎必然出现不一致。
- **修改要求**：二选一，并在文档中明确选定方案：
  - **方案 A（推荐，更懒）**：删除 `factionIds` 冗余字段，所有消费方用 `affiliations` + 时间参数推导"当前所属"。在 `lib/data.ts` 提供 `getCurrentFactionIds(person, date)` 工具函数。消除不一致的根因。
  - **方案 B**：保留冗余字段，但在 `lib/data.ts` 中提供 `normalizePerson(person)` 函数作为**唯一**写入入口，所有数据修改必须经过它，它负责同步两个字段。禁止直接修改 `affiliations`。
  - 在 §3.2 明确写出所选方案和强制规则。

### Prompt 1.2 【功能描述与数据结构矛盾】`faction-interaction` 的"双向箭头"与 `Relationship` 的有向结构冲突

- **原文位置**：§3.1 `Relationship` 接口、§4.2 边样式表
- **问题**：§4.2 边样式表里写 `faction-interaction` 是"双向箭头"，但 §3.1 的 `Relationship` 是严格有向的（`from` → `to`）。双向箭头要么需要两条有向边（A→B 和 B→A），要么需要一个真正的无向/双向结构。当前数据模型只支持前者，但文档没说清。
- **修改要求**：
  1. 明确 `faction-interaction` 在数据层的表示法：推荐用**两条有向边**表示双向互动（并在 `lib/data.ts` 的查询函数中自动合并，消费方不用手动配对）。
  2. 在 §4.2 补充：渲染时如何把两条有向边合并成一条双向箭头（Cytoscape 的 `curve-style: bezier` + 双向 `target-arrow` / `source-arrow`）。
  3. 检查其他关系类型是否有类似问题（如 `social"私人关系"是否也无向？）。

### Prompt 1.3 【功能描述模糊】时间轴"范围选择"语义不清

- **原文位置**：§4.3 `TimeAxisProps`、§4.3 `TimeState.dateRange`
- **问题**：文档提到"拖拽选择时间范围"和 `dateRange: [string, string]`，但没说清范围选择**干什么用**。在地图视图里，范围选择是显示该区间内所有事件？还是显示区间末端的控制状态？在关系图里，范围是过滤关系的生效时间段？还是高亮该时段新增的关系？不同语义对应完全不同的实现。
- **修改要求**：
  1. 明确 `dateRange` 的语义：推荐定义为"当前视图的时间过滤窗口"——地图显示该范围内的事件累积、关系图显示该范围内生效的关系、时间轴在该范围内播放。
  2. 或者，如果 `dateRange` 只是"播放起止范围"，那就改名字为 `playRange` 并缩小语义。
  3. 在 §4.3 和 §6 数据流中写明：`dateRange` 变化时，`lib/data.ts` 的哪些查询函数会收到这个参数、如何过滤。

### Prompt 1.4 【受众承诺未落地】"分层体验"只有口号没有设计

- **原文位置**：§0.3 目标受众、§5 页面结构
- **问题**：§0.3 承诺"分层体验——概览模式给初学者，深度信息给研究者"，但 §5 的每个详情页只给了一种视图，没有"概览/深度"的切换机制。这个承诺在组件设计和页面结构里完全落空。
- **修改要求**：
  1. 给每个详情页（event/person/faction）定义"概览层"和"深度层"的具体内容差异。例如：事件详情——概览层 = 一句话概要 + 时间地点 + 3 个关键影响；深度层 = 全文描述 + 所有参与者 + 完整史料出处 + 学术争议。
  2. 在组件层面定义切换机制（如 Tab、折叠面板、或"展开更多"按钮），在 §4 / §5 中写明。
  3. 如果觉得每个页面都做两层太重，至少给首页和关系图做分层，并在文档中明确哪些页面有分层、哪些没有。

### Prompt 1.5 【Phase 0 验收标准过低】"能访问空白首页"不等于基础设施就绪

- **原文位置**：§7 Phase 0 验收标准
- **问题**：Phase 0 验收标准是"`next export` 成功，GitHub Pages 能访问空白首页"。这太低了——它验证了部署链路，但完全没验证**核心可视化技术可行性**。ECharts 中国地图注册、Cytoscape 在 Next.js 里的 SSR 兼容性、Zustand 与 App Router 的集成，这些才是真正的风险点。
- **修改要求**：在 Phase 0 验收标准中增加：
  1. ECharts 能成功注册并渲染中国地图（哪怕没数据，至少省份轮廓出来）。
  2. Cytoscape 能在 Next.js App Router 中正确初始化并渲染 10 个测试节点。
  3. Zustand store 的 `currentDate` 能在两个组件间同步。
  4. 这些正是 Phase 1 的地基，Phase 0 不验证，Phase 1 会踩坑。

## P2 — 完善性（不阻塞实现，但影响产品质量）

### Prompt 2.1 【数据结构缺失】`GraphNode.style` 类型过于宽松

- **原文位置**：§3.1 `GraphEdge.style: Record<string, unknown>`
- **问题**：`style` 用 `Record<string, unknown>` 等于没有类型检查。Cytoscape 有明确的样式属性集合（`line-color`、`width`、`target-arrow-shape` 等），用 `unknown` 会让消费方写错属性名也编译不过。
- **修改要求**：定义一个 `CytoscapeStyle` 类型（至少包含 §4.2 里用到的属性），或者直接引用 `@types/cytoscape` 的 `StylesheetStyle` 类型。同理检查 `GraphNode` 是否有类似问题。

### Prompt 2.2 【搜索排序未定义】`search(query)` 的结果排序规则缺失

- **原文位置**：§6 `search(query) → SearchResult[]`、§5.3 搜索弹窗
- **问题**：搜索返回 `SearchResult[]`，但没说怎么排序。搜"林"，匹配到人物"林彪"、事件"林彪事件"、派系可能也有——什么排前面？按类型？按匹配度？按重要性？
- **修改要求**：在 `lib/data.ts` 的 `search` 函数注释中定义排序规则（推荐：精确匹配 > 前缀匹配 > 子串匹配，同等级内按重要性 `major > significant > minor` 排序）。

### Prompt 2.3 【"交流（后期）"范围模糊】Phase 4 的"评论"粒度未定义

- **原文位置**：§0.4 核心功能表、§7 Phase 4
- **问题**：Giscus 是整页一个评论区还是每个事件/人物/派系各一个？Giscus 靠 GitHub Discussions 的 `pathname` 匹配来决定评论归属，如果粒度没定义，Phase 4 做的时候还得回来改。
- **修改要求**：明确评论粒度——推荐**每个实体（事件/人物/派系）一个独立评论区**，使用 Discussions 的 `title` 或 `category` 区分。在 §4 的 `GiscusComments.tsx` 描述中写明。

### Prompt 2.4 【"内容贡献"流程与受众矛盾】GitHub PR 对主要受众门槛过高

- **原文位置**：§0.5、§5.4 贡献指南、§7 Phase 4
- **问题**：主要受众是"马列毛主义者、进步人士、历史自学者"，但内容贡献全靠 GitHub Pull Request。这个人群里会写 PR 的比例极低，等于堵死了内容贡献。
- **修改要求**：
  1. 保留 PR 作为正式入库渠道，但增加一个**低门槛入口**：在详情页加"建议修正"按钮，点击后预填一个 issue 模板（用户只需写文字，不用懂 Git）。
  2. 在 §5.4 贡献指南中同时写"低门槛入口（GitHub Issue）"和"正式入口（PR）"两条路径。

### Prompt 2.5 【"全年 5-10 个事件"的数据量矛盾】Phase 3 目标与性能目标不匹配

- **原文位置**：§7 Phase 3 "每年至少 5-10 个关键事件"、§9.2 "关系图能渲染 100+ 节点不卡顿"
- **问题**：10 年 × 5-10 事件 = 50-100 个事件。加上人物和派系，关系图总共也就 100-200 个节点。§9.2 要求"100+ 节点不卡顿"——那刚好卡在边界上，说明数据量刚好够压测。但 §10 风险表又说"数据量大导致性能问题"，**到底数据量大不大？** 如果最终就 200 节点，根本不需要虚拟滚动和复杂按需加载。文档自相矛盾。
- **修改要求**：
  1. 给出一个**内容规模估算**：Phase 3 完成时预期的事件数、人物数、派系数、关系数。
  2. 基于这个估算，判断"性能优化"是否真的是风险，还是过早优化。如果是后者，把 §10 那条风险降级，把 §9.2 的"100+"改成实际预期数字。

### Prompt 2.6 【"播放速度"单位歧义】`speed: number` 语义不清

- **原文位置**：§4.3 `TimeAxisProps.speed`
- **问题**：`speed` 标注"天/秒"，但 `1x/2x/4x` 是倍率。`1x` 对应多少天/秒？文档没说。"倍率"需要一个基准值。
- **修改要求**：明确基准速度（推荐 `1x = 30 天/秒`，即 1 秒走一个月，10 年约 120 秒播完），并在 §4.3 写明 `speed` 到实际"天/秒"的换算公式。

### Prompt 2.7 【缺少错误边界】GeoJSON 加载失败的降级策略

- **原文位置**：§3.4 GeoJSON 来源、§4.1 渲染逻辑
- **问题**：GeoJSON 要从 DataV 在线获取，但 GitHub Pages 是静态托管——**根本没有"运行时获取"的能力**。文档说"运行时从 DataV 获取，构建时缓存到 public/china.json"，但 GitHub Pages 没有 serverless function，"运行时获取"要么是浏览器端 fetch（跨域、可靠性差），要么就根本做不到。这是 §0.5 "零服务器"约束下的真实矛盾。
- **修改要求**：
  1. 明确 GeoJSON 加载策略：推荐**构建时一次性下载到 `public/china.json`**，纯静态引用，不依赖运行时网络。
  2. 删除"运行时获取"的说法（与零服务器矛盾），或明确说明是浏览器端 fetch + 失败时 fallback 到 `public/china.json`。
  3. 在 §4.1 补充：地图加载失败时的 UI 降级（显示文字提示 + 事件列表，而不是白屏）。

## 执行建议

给执行 agent 的额外提示：

1. **先改 P0，再 P1，再 P2**。P0 的答案会影响 P1/P2 的修改方向（比如 0.1 改掉 `RegionControlMap` 后，1.3 的 `dateRange` 语义要对应调整）。
2. **每条 prompt 改完后**，在原文档对应章节旁用 `> [审查修订]` 标注改了什么，方便回溯。
3. **Prompt 0.2 是预研任务**，不是纯文档修改——它需要实际去查史料。如果执行 agent 没有外部搜索能力，把这条标为"需要人工验证"并输出一份给人工的预研 checklist。
4. **保持文档其余部分的风格**：表格、TypeScript 块、章节编号沿用现有格式，不要重新发明。
5. 本文档**不删除任何原有内容**，只补充、修正、或标注需要人工决策的点。
